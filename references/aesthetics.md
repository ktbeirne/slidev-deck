# 美的ガイドライン（実装層の判断基準）

各ページに何を書くか（文言・図の構成要素・ラベル）は上流工程で確定済み。本書が扱うのは**どう見せるか**（レイアウト・配置・サイズ・色の使い方・図のスタイル）だけである。実装は1ページずつ「実装→自己検分→ユーザ提示」で進める。各規則は自己検分でyes/no判定できる問い、または具体的な禁止・数値として書いてある。

## 1. 三原則

- **主役の視覚的優位**: ページ台本の主役が3秒で目に入るか。サイズ・位置・コントラストのいずれかで最も支配的な要素にする。**問い: このページで最も目立つ要素は主役か。** 最も目立つものが装飾・脇役の図であれば失敗。
- **図の中にも焦点を1つ**: 図が主役のページでは、図の内部にも視線の着地点（構想が指定する目的地・結論の要素）を1つ定める。均等な要素の羅列で終わる図は主役不在で、読者の視線が迷子になる。焦点の強調は**塗り＋白抜きを第一選択**とする（離散的なスタイル変更で確実に効き、レイアウトを壊さない）。サイズ差による強調は補助に留める — サイズは整列・行対応・余白を巻き込む空間レイアウトの変更であり、実装時の崩れリスクが高い（強調チャネルは知覚的な強さだけでなく実装の頑健さでも選ぶ）。**構想が指定した強調は、後続の修正（行対応・整列・色統一など）で消さない** — 修正後に「焦点はまだ焦点か」を確認する。
- **引き算**: 迷ったら足すのではなく削る。余白は埋める対象ではない。装飾（3D効果・不要なグリッド線・意味のないグラデーション）は情報を運ばない要素であり、判定は「この要素は理解を助けているか」の一点のみで行う（data-ink比の考え方）。
- **一貫性**: 同じ意味には同じ形を使う。ページ間で視覚文法（配色の対応・アイコンの割当・フローの向き）を変えない。

## 2. 階層と声量（タイポグラフィ）

- 1スライド内のテキストは**主張（最大）＞根拠・ラベル（中）＞補足（小・薄）**の3段以内に抑え、この差をはっきりつける。全文が同じサイズ・同じ太さで並ぶと、読み手はどこが重要か分からない。
- サイズは `--text-xs` 〜 `--text-display` のトークンから選ぶ。中間の独自サイズを作らない。
- そのスライドの結論一文には `.takeaway` を使う。**1枚に1つまで。**
- 補足・注記・出典は `.dim` で明示的に声量を下げる。
- 重要語の強調は `<b>` または `.accent`。**1ブロックに1〜2語まで。** 全部を強調すると何も強調されない。
- **末尾の孤立改行を作らない**: ボックス・カード内の一文が、末尾1〜3文字だけの行（例: 「す。」のみの行）で折り返すのは失敗。対処は幅の調整、`text-wrap: balance`（短文の行長を自動で均す。Chromium対応済みでSlidevのレンダ・exportの両方で効く）、意味の切れ目への明示的な `<br>`。文言を短くして解決したい場合は肉付けを先行更新する（実装側で勝手に文言を変えない）。
- **収まるなら1行**: 見出し・ラベル・ボックス内の短文は、無理なく1行に収まるなら1行にする。余白が十分あるのに2行へ折るのは体裁の緩みに見える（改行は幅の必然か意味の区切りがあるときだけ）。複数文のキャプション・従属文も同様に1行が既定。ただし**2行に割る意図（対句のリズムを立てる・行長が35〜40字を超える等）を1行で言えるなら2行でよい** — これは規則でなくトレードオフ判断なので、意図の説明責任だけを課す。言えないなら1行（片側だけ大きく余る改行は間延びに見える）。
- **用語を成り行きで分断しない**: 用語・製品名がコンポーネント幅に収まらないときは、**意味的区切り＞文字数均等**の優先度で改行位置を自分で決める（例: 「Claude Sonnet」が1行に収まらないなら「Claude<br>Sonnet」。幅の成り行きで「Claude So / nnet」のように割れるのは不可）。用語には `white-space: nowrap` か明示 `<br>` を使い、ブラウザの自動改行に任せない。日本語の複合語も語境界で切る（「導入スケジュール」なら「導入 / スケジュール」）。

## 3. 色の規律

- 中立色（`--surface` / `--text-color`）が視覚面積の6〜7割を支配し、`--brand-primary` は「ここぞ」のアクセントに絞る。全色を均等に使わない。支配率は**ページ全体の面積で見る目安**であり、主役の図の内部にまで単色を強制しない。
- `--accent-info` / `--accent-warn` / `--accent-success` / `--accent-neutral` の4色は、既定の意味（情報・注意・成功・中立）での使用を基本としつつ、**図内の役割の塗り分け**にも使ってよい（例: 成果・目的地のボックスだけ強調色にする、ひとつの仕組みの層・段階を同系の濃淡で描き分ける）。条件は3つ: **同じ概念には同じ色**（デッキ内で一貫させ、途中で意味を変えない）、**1つの図で中立以外は3色相まで**（同一色相の濃淡は1色と数える）、**色だけに意味を載せない**（ラベル・形状を併用）。目的を1行で言えない装飾色は引き続き禁止。
- **塗りつぶし＋白抜き文字は最上位の強調**: 面で塗った上に白抜き文字を置く形は、輪郭線スタイルより視覚重量が一段強い（UIのボタン階層の contained=primary / outlined=secondary と同じ）。1ページ・1図の中で塗り＋白抜きの役は**1系統まで**に絞り、対等な要素群の中で塗りと輪郭を混在させない（同形の塗り/輪郭差は選択状態・重要度差の含意を生む）。白抜きは短いラベル・見出し専用: 長文を白抜きにしない、細いウェイトを使わない（ハレーションで潰れる）。大面積の高彩度塗りは面積効果で支配率を壊すため、塗りは小さい要素（バッジ・チップ・帯）か1枚の主役帯に限る。
- **濃淡は同一概念内の描き分けにのみ使う**: 同一色相の濃淡差は「これらの要素には関連がある」という視覚的示唆を持つ（レイヤーの違い、全体と部分、包含・親子、強弱）。この用途にだけ使い、**対等な別概念の対比に濃淡の対を使わない** — 対等な別概念は同一色か、明確に別の色相で描く（例: MCPとSkillの対比を同色の濃淡で塗ると「SkillはMCPの一種」という誤った関連が示唆される）。濃淡を使うときは、ひと目で意図的な差と分かる明度差をつける（僅差は配色ミスに見える）。同じ理論の裏面として、**順序・段階・強弱の軸に色相の切り替えを使わない**（色相は分類の変数であり順序を持たない。段階は明度・サイズ等の順序変数で描く）。
- 濃色背景（バー・帯・カード）の上に文字を置くときは、必ず明示的に `color`（`--text-inverse` 等）を指定する。継承任せにしない。**理由:** Slidev/UnoCSSのリセットやテーマ継承で文字色が意図せず白落ちするバグは、1つのデッキ内で複数箇所に独立して再発しやすく、いずれも継承任せの箇所が発火点になる。
- 色・角丸・影・余白の値は必ずトークン経由（`var(--...)`）で書く。hex値・生の px を直接書かない。**理由:** トークンにしか値が存在しなければ、微小な値のドリフト（例: 7pxと8pxの角丸が並存する状態）は構造的に起こらない。目視では発見できないため、経路を1本に絞る。**例外:** SVG の viewBox 内部単位（font-size・stroke-width・rx 等）は var() が効かないため生値を許容する。ただし色は CSS クラス経由で塗り、角丸等はトークンと同値に揃える。

## 4. 余白と整列

- 本文が構造物（リスト・グリッド・図・表・カード群など、複数要素から成るレイアウト）を持つスライドは `align: top`。主張文や連続する地の文だけのスライドは、段落が複数あっても `align: center` でよい（対象は構造物の有無であり、地の文の行数ではない）。迷ったら両方を実レンダし、上詰まりが出ない方を選ぶ。
- **問い: `align: top` で本文が画面の上半分だけで終わり、下が大きく空いていないか。** 空いていれば「上詰まり」で間延びして見える失敗。対処は次の優先順で検討する。
  1. 単一構造（FlowSteps 1本等）のスライドは、まずコンポーネント自体の拡大（フォントサイズ・パディング・箱の高さ）を検討する。
  2. それでも余るなら `align: center` への切替、または隣接スライドとの統合を検討する。
- 本文の中央揃えは使わない。中央揃えが許されるのは表紙タイトルと1〜2行の大メッセージのみ。本文・リストは左揃え。
- **対比構造の均衡**: 対比ボックス（Compare・2列カード等）は文章量が視覚的に同量に見える範囲で揃える。片方だけ内部余白が大きいと、対比そのものが崩れて見える。対処は (a) 対比軸ごとの行対応レイアウト（同じ行に同じ属性が並ぶ形）に変える (b) 文章量の偏りが原因なら肉付けの文言を先行更新で揃える。フォント拡大や無意味なパディングで空白を糊塗しない。
- プロセス図・対応関係図は、目分量の flex 調整ではなく**意味グリッド**（時間軸・段階を CSS grid の列に対応させる）で組む。**理由:** 目分量で組んだフロー図は要素数やラベル長が変わるたびに崩れるが、列線に意味を固定すれば崩れない。
  - コンテナに `grid-template-columns: repeat(N, 1fr)` を与え、フェーズ `i` を対応する列範囲（`grid-column`）に配置する。列数は等分なら「フェーズ数×2」を目安にする。
  - 複数レーン（例: 上段が作業、下段が成果物）がある場合、**同じ列線を全レーンで共有する**。レーン間で列番号がズレると意味的対応が崩れる。

## 5. 図の規律

- フローの方向は全図で統一する（左→右 または 上→下）。図ごとに向きが変わると読み手の学習が毎回リセットされる。
- 独自の記号・色分けを使うなら凡例を必ず付ける。色だけに意味を持たせない（ラベル・線種を併用し、色覚多様性に対応する）。
- 繰り返し登場する概念（人・AI・サーバ・ドキュメント等）にはアイコンを1対1で割り当て、デッキ全体で固定する。**問い: 同じ概念が図によって違うアイコンで出ていないか。** 出ていれば読み手は毎回読み直しになる。
- アイコンは意味の符号化にのみ使う。単独に意味を持たせず、初出時はラベルを併記する。1スライドに種類を詰め込みすぎない。
- **同属性の要素は同サイズ**: 図内で同じ属性の要素は原則同じ幅・高さにする（フローチャートなら処理ノード同士は同サイズ、分岐ノード同士は同サイズ）。文字数に応じた成り行きサイズにせず、共通クラスやグリッドで寸法を固定する。サイズの差は属性の差を表すときにだけ使う（差が意味を持つ）。
- **対称な関係は対称に結線する**: 二項関係の記号（＋・＝・⇄ 等）は両側の要素と対称に接続する。片側だけ線で結ぶと、結線はグルーピングの信号なので「その記号は片側にだけ属する」ように読まれる。両側とも結ぶか、両側とも結ばないかの二択。
- **矢印は順序・因果・フローにのみ使う**: 矢印の向きは「時間的順序・因果・処理の流れ」の主張として読まれる。順序のない関連・参照・対等な依存を一方向矢印でつなぐと、存在しない処理順序を読み取られる。順序が無い関係は矢じり無しの線か双方向で描く。**問い: この矢印の始点→終点には実際に順序・因果があるか。**
- **アイコンは既成セットから**: アイコンを自分で SVG パスとして描かない。ライセンスが明確なアイコンセット（MDI = Apache-2.0、その他 MIT / CC0 等。Slidev では iconify 経由）から選ぶ。**理由**: 既成セットはデザイナーの成果物で造形品質と統一感が担保されている。自作パスはサイズ・線幅・造形が揃わず、図全体を素人くさく見せる。
- **アイコンの種は概念と一致させる**: 人型アイコンは人（ユーザ・社員）にのみ使う。AI・システム・仕組みに人型を当てない（読者の概念対応が崩れる）。
- SVG矢印は `preserveAspectRatio="none"` で引き伸ばさない。引き伸ばすと矢じり・線幅まで一緒に歪む。矢印は viewBox 固定で作るか、単純な三角矢印は CSS の border 三角形で代替する。可変長の矢印が必要な場合、線部分（伸縮）と矢じり部分（固定サイズ）を分離して実装し、1つの図形に混在させない。

## 6. 同質性の回避

- 同一の視覚形式（箇条書き・カード群・FlowSteps・Compare 等）を**3枚以上連続させない**。
- FlowSteps・Compare・カード群はいずれも「角丸の箱＋矢印」で描かれるため、これらだけを並べると形式を変えても全ページ同じ見た目になる。**問い: 直近3枚が全て箱系（カード・パネル・フロー）だけで構成されていないか。** そうであれば、アイコン・大数字・実物スクリーンショット・表・余白で見せる大メッセージのいずれかを混ぜる。
- 箇条書きは最終手段。図・表で示せる内容を箇条書きで代替しない。均等な長さの3〜4項目を機械的に並べる形は生成AIの指紋であり、対比なら表か Compare、流れなら FlowSteps、構造なら図に変換する。箇条書きが正当なのは「他の型に変換できない、真に並列な列挙」だけ。

## 7. AIスロップの指紋（負のリスト）

見た目に関して、生成AIのスライドに共通して現れ、読み手に「雑な資料」という印象を与える特徴。

| 指紋 | 対処 |
|---|---|
| タイトル上下のアクセント線・装飾カラーバー | 使わない |
| 本文の中央揃え | 本文・リストは左揃え。中央揃えは表紙タイトルと大メッセージのみ |
| 均等な長さの3〜4項目箇条書きを毎スライド生成 | 6章のとおり視覚形式へ変換する |
| 意味のない番号・アイコン装飾 | 番号（01/02/03）や区切り線は、内容が実際に順序・区分を持つ場合のみ使う |
| 同一レイアウトの3枚以上連続 | 6章のとおり視覚形式を切り替える |
| 全ページが「角丸の箱＋矢印」の同質構図 | 6章のとおり箱以外の視覚語彙を混ぜる |
| 全要素が同じ声量 | 2章のとおり3段の声量差をつける |
| 定番AI配色（クリーム背景＋テラコッタ、紫グラデーション、均一角丸） | パレットはテーマの semantic トークンのみ使う。装飾色を足さない |

## 8. 提示前検分チェックリスト

1ページ実装し終えたら、ユーザ提示の前に上から順に自問する。

1. このページで最も目立つ要素は主役（heading/takeaway）か。
2. 削れる装飾・余白の埋め合わせが残っていないか。
3. 声量は3段以内で、差がはっきりついているか（`.dim` で下げるべき補足が地の文と同格になっていないか）。
4. hex値・生の px・生の radius/shadow を直接書いていないか（全てトークン経由か）。
5. 濃色背景の上の文字に `color` を明示したか。
6. `align: top` で下側が大きく空いて上詰まりになっていないか。
7. 本文・リストが中央揃えになっていないか（表紙・大メッセージを除く）。
8. 意味グリッドを使うべき図で、目分量の flex 配置になっていないか。
9. 直近3枚のレイアウト形式・箱系の見た目が連続していないか。
10. 図の凡例・アイコンの概念対応が、これまでのページと矛盾していないか。
11. ボックス内の文が末尾1〜3文字だけの行で折り返していないか。用語が成り行き改行で分断されていないか。
12. 対比ボックスの文章量・余白は釣り合っているか。図内の同属性ノードのサイズは揃っているか。並列カードの内部要素（アイコン・見出し・補足・例）の行位置とセンタリングはカード間で揃っているか。
13. アイコンは既成セット由来で、サイズが揃い、種が概念と一致しているか（人型がAI・仕組みに使われていないか）。
14. 矢印は順序・因果・フローにのみ使われているか（順序のない関連が一方向矢印になっていないか）。順序・段階の軸に色相の切り替えを使っていないか。

## 9. 理論バックログ（実例待ち）

確立された理論に由来するが、本ワークフローの実走でまだ事故の実例が出ていない候補規則。実例が出たら該当章へ昇格させる（規則は実例からのみ増やす原則）。検分チェックリストの対象ではない。

- **近接の誤用**（ゲシュタルト・近接）: 無関係な要素を余白調整の都合で近づけると関連があると誤読される。関連要素の間隔＜無関係要素の間隔、を崩さない。
- **共通領域の誤用**（Palmer & Rock）: 囲み・背景色はそれ自体が「同じグループ」という主張になる。同列要素の一部だけを囲んで、意図しない特別扱いに見せない。
- **前注意属性の同時強調の上限**（Ware）: 主役の強調に使う視覚属性（色・サイズ・太さ・影）は同時に2つまで。全部盛りは強調を消す。
- **行長の上限**（組版慣行）: 日本語本文の1行は35〜40字を目安とし、画面幅いっぱいに間延びさせない。

（保留判断済み: 共通運命=静止画に不適用／lie factor=データチャート層の別規則／layering=声量3段と重複／連続=フロー向き統一と重複／Z・F型読線=主役規則で個別処理の方が実効的）
