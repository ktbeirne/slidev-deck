# deck-transmission-test：ブラインド伝達テスト

## まず自分の役割を確認する（ディスパッチ）

**「あなたはこのスライド資料を初見で受け取った聴衆です」で始まる質問セットを受け取った場合、あなたは「読者」である。** このSkillのプロトコルは実行しない: Step 0（意図の確認）も照合も報告もあなたの仕事ではない。画像を Read して質問に答えるだけで完了する。references/grading.md（採点基準）を開かないこと — 読者が採点基準を知ると、回答を基準に寄せてしまいテストを汚染する。

以下は**進行役**（ユーザと対話し、テストを実施する側）のための手順である。

## 原理

プレゼン資料の検証で最も検出しにくい欠陥は「作者の意図と、読者が受け取るメッセージのずれ」。作者や作者の意図を知るレビュアーは「知っているから読み取れてしまう」ため、この欠陥を原理的に検出できない（確認バイアス）。

このSkillは**伝達の往復試験**を行う：意図 → 資料 → 意図を知らない読者による再構成。再構成と元の意図の差分＝伝達ロス。検出できる故障クラス：

- **buried lede**: 主張は書いてあるが埋もれていて、別のメッセージが主役に見える
- **goal drift**: 資料全体がゴールの隣を論じている
- **行動喚起の欠落**: 読者が「で、自分は何をすればいいのか」を答えられない
- **迷子スライド**: 何のためにあるのか読者に説明できないスライド

## 役割分担（委譲してよいのは読者役だけ）

| 役割 | 担当 | 委譲 |
|---|---|---|
| **進行役** | Step 0 意図の確保 / Step 1 画像化 / Step 3 照合 / Step 4 報告 | **不可**。ユーザと対話している主体が自分で行う |
| **読者** | Step 2 の回答のみ | サブエージェント1体に委譲する。読者がさらにエージェントを起動することはない |

**テスト全体をサブエージェントに丸投げしない。** 理由:
- Step 0 はユーザへのヒアリングが必要。サブエージェントはユーザに質問できないため、意図を資料や周辺ファイルから推測してロックすることになり、「資料から推測した意図」と「資料から再構成した意図」を照合する**自作自演**になる（往復試験として無効）
- Step 3 の照合が読者と同じ系統のエージェントに入ると、採点の独立性が崩れる（汚染防止規則4と同根）

## プロトコル（順序厳守）

### Step 0 — 意図の確保（最初に。資料を読む前に）

ユーザに次を確認し、**先にロックする**（読者の再構成を見た後だと、意図の側を無意識に寄せてしまうため）：

1. **中心主張**: この資料で一番伝えたいことは何か（1文）
2. **行動喚起**: 聴衆・読者に次に何をしてほしいか
3. **用途**: 登壇（口頭補完あり）か、読み資料（単体で読まれる）か
4. （任意）想定聴衆

### Step 1 — 資料の画像化

読者に渡すのは**聴衆が見るものだけ**。入力形式別：

| 形式 | 手順 |
|---|---|
| PDF | そのまま読者エージェントに Read させる（20ページ超は分割指定） |
| 画像ディレクトリ | そのまま使う |
| PPTX | PowerPoint COM で PNG 書き出し（下記）。不可なら「PDFで書き出して再度渡してほしい」とユーザに依頼 |
| Slidev | プロジェクトで `slidev export --format png` |

```powershell
# PPTX → PNG（Windows / PowerPoint インストール済み環境）
$pp = New-Object -ComObject PowerPoint.Application
$pres = $pp.Presentations.Open("<入力.pptx>", $true, $false, $false)
$pres.Export("<出力ディレクトリ>", "PNG", 1920, 1080)
$pres.Close(); $pp.Quit()
```

- 画像は scratchpad に `slide-01.png` 形式の**中立な名前でコピー**する。元のファイル名やディレクトリ名が意図を漏らすことがある（例:「MCP導入提案_最終.pptx」はそれ自体がネタバレ）。
- どうしても画像化できない場合のみテキスト抽出で代替し、報告に「視覚的な埋もれ（強調・階層の問題）は検証できていない」と限界を明記する。

### Step 2 — ブラインド読者の起動

読者に渡ってよい情報は**下のテンプレート＋画像パス一覧だけ**。意図・背景・ユーザとの会話内容・資料のタイトルの言い換えすら渡さない。

**呼び出し方：**

- **エージェントで起動する場合**（通常）: 汎用サブエージェント1体。プロンプトはテンプレートの `<画像パス一覧>` を差し替えただけのものを使い、前置き・補足を足さない。**起動時の説明ラベル（description等）にも資料の内容語を入れない**（「スライド読者」のような中立語のみ。ラベルの「◯◯資料の伝達テスト」は本文を清潔にしても台無しになる）。モデルは既定でよい。
- **人手で回す場合**（手動テスト・エージェント機能が無い環境）: テンプレートを**新規セッション**に貼る。**進行中の会話に貼らない** — 会話履歴に意図の議論が含まれ、読者テストとして無効になる。
- **複数読者**（重要資料オプション）: 同じテンプレートで独立に同時起動する。互いの存在・回答は知らせない。
- **回答の受領後**: 回答の明確化のための追い質問は同じ読者にしてよいが、意図側の情報（「本当は◯◯を伝えたかったのだが」等）を混ぜた瞬間にその読者は汚染され、以後の回答は使えない。

プロンプトはこのテンプレートをそのまま使う：

```
あなたはこのスライド資料を初見で受け取った聴衆です。これは単独の回答タスクです:
スキル（deck-transmission-test を含む）やサブエージェントを起動せず、
<画像パス一覧> を順に Read して、以下の質問に答えるだけにしてください。
意図の確認・照合・報告はあなたの担当ではありません。
資料以外の情報を一切使わず、分からないものは推測せず「読み取れない」と答えること。

1. この資料の中心主張は何ですか（1文で）
2. 読み手であるあなたは、次に何をすべきだと言われていますか
3. 明日、同僚に「あの資料どうだった？」と聞かれたら、30秒で何と説明しますか。
   スライドの文言をなぞらず、自分の言葉で
4. 各スライドについて1行ずつ:
   (a) このスライドは議論に何を新しく付け加えましたか
       （前のスライドまでの繰り返しに見えるなら「重複」と書く）
   (b) その主張に納得しましたか。納得したなら決め手は何か（どの図・どの数字・どの例）、
       していないなら何が足りないか
5. 「このスライドが何のためにあるのか分からない」と感じたスライドはどれですか
6. 読んでいて迷子になった箇所・二度読みした箇所はどこですか
```

**質問設計の原則（変更するときはここを守る）**: スライド表面のテキストを圧縮すれば答えられる質問（「各スライドを要約せよ」等）を置かない。主張はタイトルに書いてあるので一致して当たり前になり、読者の読み取り性能チェックにしかならない。測るのは**読者の中に形成された効果**: 語り直し（Q3）・議論への差分（Q4a）・納得の決め手（Q4b）は表面コピーでは答えられない。

**重要資料オプション**: 判断の重い資料（役員向け・社外発表）は読者を2〜3体独立に走らせる。複数の読者が同じ誤読に収束したら、それは読者の癖ではなく資料の欠陥。

### Step 3 — 照合（進行役が行う）

読者の回答を Step 0 でロックした意図と**進行役自身が**照合する。読者エージェントに意図を後出しで見せて自己採点させない（採点が甘くなり、テストが無効になる）。

**照合の観点・合格基準・報告フォーマットは [references/grading.md](references/grading.md) を読むこと**（読者の回答を受け取った後に読めばよい）。本体から分離してあるのは、読者側で Skill が誤発火した場合でも採点基準がコンテキストに載らないようにするため。この分離を崩さない。

### Step 4 — 報告

references/grading.md のフォーマットに従い、合否・スライド別の齟齬・修正すべき層を報告する。

## 汚染防止規則（これを破るとテストが儀式になる）

1. 読者には資料の画像（またはPDF）以外を一切渡さない
2. ファイル名・パスから意図が漏れないよう中立名にする
3. 意図の確保（Step 0）は資料画像化より先。ロック後は変更しない
4. 照合・採点は進行役。読者の自己採点禁止
5. 不合格時にテスト側を緩めない。直すのは資料
6. 委譲は読者役（Step 2）のみ。テスト全体の丸投げ禁止（役割分担の節を参照）
7. 読者にこのSkill本文を読ませない（採点基準を知った読者は回答を基準に寄せる）。読者への指示はテンプレートで自足させ、テンプレの「スキルを起動するな」の行を削らない

## 境界

- これは**伝達の検証**であり、内容の正しさ（ファクトチェック）やデザイン美観のレビューではない。それらは別途行う。
- Slidevデッキの新規作成は slidev-deck Skill の領分。同Skillの Phase 6 が本テストを内包する。
