# slidev-deck: スライドの段階構築

原稿からいきなりスライドを書かない。5つのフェーズを順に進め、各フェーズの終わりで**確認ゲート**（成果物を提示してターンを終え、ユーザの確認を待つ）で合意してから次へ進む。一括生成は系統的な崩れを最後に露見させ、手戻りが全ページに波及する — 段階化とゲートはその構造的な防止策。

| Phase | 内容 | 成果物 | ゲート |
|---|---|---|---|
| B | ブリーフ確立 | deck-brief.md | 全項目の確定 |
| T | デザイントークン確立 | theme.css＋フレーム選択 | specimen でパレット承認 |
| S | ストーリーボード（絵コンテ） | storyboard.md | 構成の合意 |
| C | 肉付け（ページ台本） | storyboard.md に追記 | 文言の最終確認 |
| I | 実装（1ページずつ） | slides.md ほか | ページ毎の確認ループ |

## 全フェーズ共通の原則

- **確認ゲートで必ず停止する**: 提示してターンを終える。合意を得る前に次フェーズの作業を始めない。
- **先行更新の原則**: 上流の成果物が正。下流で変えたくなったら、上流ファイル（brief → storyboard → 肉付け）を先に更新してから下流へ反映する。
- **発明ゼロ**: 下流工程は上流の文言・数字・ラベルを転記する。発明が必要になった時点で上流の記述不足なので、上流に戻る。
- **状態のファイル外在化**: 途中で確立した取り決め・変更は必ずファイル（deck-brief / storyboard / style-notes）に記録する。報告文や会話にだけ残した取り決めは、後続を別セッション・別エージェントが実装する時点で消滅する。**後続はファイルしか読めない前提で書く**。

## Phase B: ブリーフ確立（確認ゲート）

**目的**: デッキの前提条件を1枚に固定する。ここで決まる値は見た目（Phase T の配色方向）だけでなく、語り方・密度規範・検証の照合先まで全工程が参照する。トークンより前に独立ゲートとするのは、判断軸が「内容の前提」であり「見た目の好み」と混ぜると議論が濁るため。

**手順**:

1. ユーザの入力（原稿・依頼文）を読み、下の書式で `deck-brief.md` のドラフトを作る。読み取れる項目は埋め、読み取れない項目は**推測で埋めず「要確認」と明記**する。
2. **入力資料の受領と目録化**: 受け取った資料（原稿md・既存資料・データ・スクショ・参考リンク等）を「入力資料」欄に列挙し、それぞれ何に使えそうかを1行で仮置きする。「メッセージ上あった方がよいが手元にない素材」に気づいたら「依頼」として挙げる（不足素材の依頼は絵コンテ段階でも再度行う）。
3. ドラフトを提示し、**ターンを終えて確認を待つ**。全項目が確定するまで次へ進まない。
4. 確定した deck-brief.md をデッキプロジェクトに保存する。途中で前提を変えるときは先にこのファイルを更新する。

**deck-brief.md の書式**:

```md
# デッキブリーフ
- テーマ: <何の資料か。1行>
- ゴール: <聴衆に最終的に何が起きれば成功か。1文（源流ゴール）>
  - 確認基準: <達成をどう見るかの簡易チェック。箇条書き可。基準は源流ゴールの従属物であり、資料の章立てや目的そのものではない>
- 聴衆: <誰が見るか。前提知識の水準>
- 用途: 配布 | 登壇 | 報告
- トーン: フランク | 標準 | 固い
- 想定ページ数: <N>（表紙除く）

## 入力資料
- <パスまたは名前>: <種別（原稿/データ/スクショ/参考）と使い所の仮置き>
- 依頼: <欲しいが手元にない素材。なぜ要るかを1行で>
```

**各項目の連動先**（この表があるからブリーフが飾りにならない）:

| 項目 | 下流での使われ方 |
|---|---|
| テーマ・ゴール | 絵コンテの主役メッセージの親。伝達テストの照合先。**構成の重心と導入は源流ゴールに従う**。確認基準は到達点の検証に使い、章立てに直訳しない |
| 聴衆 | 用語の展開度合い・前提説明の量・図の抽象度 |
| 用途 | 登壇=口頭補完あり（文字を絞る・ページ参照禁止）／配布=単体で読まれる（各スライド自足・ページ参照許容）／報告=結論先出し・ページ参照許容。deckMode に写像 |
| トーン | 文体（口語⇔である調）と Phase T の配色方向の推奨。強制はしない |
| 想定ページ数 | 絵コンテの枠数上限。超過時は枠を増やすのではなくメッセージを統合する |
| 入力資料 | 事実の出所。ドメイン固有の事実は資料からのみ引き、一般知識は具体案として書き込む（詳細: [references/claims-and-language.md](references/claims-and-language.md) 1章） |

## Phase T: デザイントークン確立（確認ゲート）

**目的**: 見た目の議論をデッキ実装の前に済ませ、以後の全スライドを theme.css 1ファイルの差し替えで統制する。色を後から直すと全スライドに波及するため、ここを最初の見た目ゲートにする。

**手順**:

1. deck-brief.md のトーン・聴衆・用途から配色の方向性を決める。
2. テンプレート一式（`template/` の内容）をデッキプロジェクトへコピーし、デッキプロジェクトで `npm install` を実行する（テンプレートに node_modules は含まれない。package.json / package-lock.json が依存を固定している）。specimen.html はトークンを読んで描画するだけの**固定ビューア**であり、デッキごとに書き換えない。
3. パレット候補を2〜3個 `themes/candidate-*.css` に生成する。[references/tokens-and-frames.md](references/tokens-and-frames.md) の**セマンティック変数契約を全変数埋める**。各候補の冒頭コメントに「狙い」と、あれば「用法上の注意」を1〜2行で書き、`--candidate-label: "<短い呼び名>"` を定義する（specimen が「表示中」ラベルとして拾う。タブ名は中立の候補A/B/C固定で、ビューア側には呼び名を書かない — 候補はデッキ毎に生成される成果物であり、固定ビューアに書くと実体と乖離するため）。**hex を書いてよいのは themes/*.css と theme.css だけ**。
4. **自己検品（提示前に必須）**: `scripts/shoot-specimen.ps1 -Theme themes/candidate-X.css` で各候補を撮影し、PNG を Read して (a) コントラスト表に「不足」が無い (b) 疑似スライド・フレームに崩れが無い、を確認する。不合格品をユーザに見せない。
5. specimen.html のパスを提示し、**ターンを終えて承認を待つ**。判断してもらうこと: 採用パレット／フレームプリセット（5種から1つ）／値の修正指示。
6. 承認された候補を `theme.css` へコピーし、冒頭コメントに「採用候補名・承認日・ユーザ修正の内容・フレームプリセットの選択」を記録する（実装層がここを参照する）。

**このフェーズで決まらないこと（後段の責務）**: レイアウト・コンポーネント・スライド構成。疑似スライドは印象確認用の近似であり、実寸・改行・フォント描画の最終確認は Phase I 手順0 の初回レンダで行う。

## Phase S: ストーリーボード（絵コンテ）（確認ゲート）

**目的**: ユーザとの中間言語として `storyboard.md` を作り、ページ構成・各ページで伝える内容・どう伝えるかをラフに合意する。詳細な要素設計はこの段階では当てにならないため作り込まない（Phase C の責務）。

**構成の作り方 — ゴールから逆算する**: 聴衆が現状から deck-brief.md のゴールに到達するために越えるべき理解のステップを列挙し、各ステップを1ページに割る。**原稿の章立てをなぞらない** — 原稿は素材であって構成の正ではない（章立てから作ると素材メモの切り貼りに堕ちる）。

**手順**:

1. 入力は deck-brief.md と入力資料。
2. **読者と到達点の詳細化**: storyboard 冒頭に次を書く。ブリーフの聴衆・ゴールの具体化であり、矛盾が出たらブリーフ側を先に直す。

```md
## 読者と到達点（構成の前提）
- ゴール設定の認識: <ゴールの構造の理解: 源流ゴールは何か／各確認基準はなぜ基準に入っているか（背景の文脈は、どの基準に効く話かを紐づける）。ユーザ入力に根拠がなければ推測であることを明記>
- ペルソナ: <具体的な1人として書く（役割・前提知識・今の認識や誤解・読む状況）。「〜な人々」と平均化しない。前提知識・誤解は根拠のあるものだけを置く>
- 到達点: <読み終えた読者が「できる」ことを検証可能な形で列挙>
- ギャップ: <ペルソナの現在地と到達点の間にある理解の段差を列挙。これがページ分割の元>
```

   「ゴール設定の認識」を最初に書くのは、ここが誤っているとペルソナ・ギャップ・構成が連鎖的に狂うため。推測が外れていても、見える形で書いてあればゲートで正せる。
   **動機の昇格禁止**: 個別の確認基準に付随する動機・文脈を、資料全体の目的や導入のフレームに昇格させない。構成の重心・導入・ペルソナは源流ゴールに従い、個別動機はその基準を扱うページの中で処理する。ペルソナを個別動機の該当者に狭めない。

3. **ギャップからページへ**: ギャップを越える理解のステップを順に並べ、1ページ1ステップで割る。
   - **提出する正の案は必ずページ予算内**。統合で品質が損なわれると考える場合は、超過版を「代替案」として理由つきで添えてよいが、正として出さない（予算はユーザが確定した制約であり、破る判断はユーザのもの）。
   - **ギャップ→ページの対応表を storyboard に含める**（構成根拠をゲートで議論できる形にする。導入・締めなどギャップに紐づかないページは役割を書く）。
   - **導入・締めページの型**: 自由作文にせず、資料の種別から型を選んで宣言する。導入 — 報告・決裁 = エグゼクティブサマリ／解説・教育 = モチベーション→ゴール→目次／登壇 = フック＋アジェンダ程度。締め — 報告・決裁 = 依頼・次アクションの明示／解説・教育 = 要点再掲＋到達点の自己チェック。型はゴールの再言明であり、新規の主張や共感演出を足さない。
4. 各ページを次の書式で書く（これ以上作り込まない）:

```md
## <番号>. <タイトル>
- 伝えること: <このページで聴衆に起きるべき理解の変化。主張文で1〜2文>
- どう伝えるか: <ラフな方針1行。例: 比較表で / 実物スクショで / 数字1つを大きく / 流れ図で>
```

   - **事実の書き分け・増幅禁止・不利な事実の扱い**: [references/claims-and-language.md](references/claims-and-language.md) 1〜3章に従う。要点: ドメイン固有の事実は資料からのみ・一般知識は具体案として書き込む／資料の記載より強い断定に増幅しない／不利な事実は隠さず盛らず、強調するかは目的判断。
   - **タイトル方針**（storyboard 冒頭で宣言し、デッキ全体で統一）:
     - **主張型**: タイトル＝ページの主張文。説得・意思決定・報告で推奨（タイトルを追うだけで結論が伝わる）。
     - **題目型**: タイトル＝内容ラベル。解説・教育・リファレンスで自然なことが多い。紙面上の主張は takeaway 行が担う。
     - どちらでも**「伝えること」は主張文で書く**（メッセージ連鎖検証の対象）。1ページ1メッセージは共通の不変条件。主張がどこにも明文化されないページを作らない。
     - 統一が既定。ただし**山場1枚に限り**題目型デッキ内で主張型へ格上げしてよい（方針宣言に逸脱箇所と理由を明記。無断の混在は不可）。
     - 表紙は方針の対象外。締めページの行動要求文は主張型デッキで許容。いずれも方針宣言に一言書く。
   - **作図構想の委託**: 型が無い視覚ページ（図・実例）は、自前の形式カタログから選ばず、隔離した構想エージェントに設計させる。[references/figure-commission.md](references/figure-commission.md) の定型プロンプト・検収基準（図だけテスト）・構成作家の権限（差し戻しと軽微修正のみ）に**厳密に**従う。
5. **理想証拠チェック（依頼の導出）**: 各ページ、最低でも山場について「ゴール達成のためにこのページに置くべき最強の証拠は何か」を**手持ち資料と無関係に**1行で考え、手持ちと突き合わせる。無いものは「依頼」に挙げる。**「手持ちの資料で組めた」は「依頼なし」の根拠にならない** — 依頼は理想の構成との差分から導出する。
6. **ゴーストデッキ自己検証（提示前・委譲禁止）**: メッセージ連鎖（主張型はタイトル列、題目型は「伝えること」列）を通読し、ストーリーが成立し**到達点の全項目が回収されるか**確認する。主張型は先に各タイトルが主張文か（題目・疑問文が混ざっていないか）を1枚ずつ確認する。入力資料があるデッキでは**増幅チェックを操作化して行う**（[references/claims-and-language.md](references/claims-and-language.md) 2章の手順）。
7. **独立レビュー（オプション・二段構え）**: 決裁・社外・重要報告では実施が既定。実施する場合は [references/independent-review.md](references/independent-review.md) に従う（理想構成レビュー＋ペルソナRPレビュー）。
8. storyboard.md を提示し、**ターンを終えて合意を待つ**。独立レビューの結果（差分・想定質問・不採用指摘と理由）を添え、「依頼」への回答（提供／無しで進める／取りに行く）をこのゲートで回収する。
9. 合意後の構成変更は、スライドを直接いじる前に必ず storyboard.md を先に更新する。

## Phase C: 肉付け（ページ台本）（確認ゲート）

**目的**: 絵コンテの「どう伝えるか」を具体化する。**完了条件 = 完成後のそのページに何が書かれているかが自然言語で説明されている状態**。判定テストは**文言発明ゼロ**: この記述だけを渡された実装者が、新しい文・数字・ラベルを一切発明せずにページを作れること（レイアウト・配色・寸法だけが実装層の裁量として残る）。

**手順**:

1. 入力はゲート合意済みの storyboard.md、deck-brief.md、入力資料、ゲートで回収した「依頼」への回答。
2. storyboard.md の各ページに `- 肉付け:` ブロックを**追記**する（ラフ行は消さない — 構成合意の記録として残す）:

```md
- 肉付け:
  - takeaway: <ページ最上位に置く主張の実文言1文。題目型デッキでは紙面上の主張をこれが担う>
  - 主役: <1つ。図なら種類・構成要素・ラベル・関係・強調点を完全に言語化。表なら列見出しと行の実テキスト。文なら実文言>
  - 従属: <最大2つ。同様に完全に言語化。無ければ「なし」>
  - 注記・出典: <実文言。無ければ「なし」>
```

3. **制約**（文言が確定する層なので、文言系規則の本番はここ）:
   - **空間予算**: 1ページ = 主役1つ＋従属2つまで。入らない要素は削るか、絵コンテに戻ってページ分割を再交渉する（勝手に詰め込まない）。予算は**紙面ブロックの数**の規律であり、図の**内部要素数を縛らない**（図内の多寡は図だけテストと美的規律で判断する。予算を図内要素に誤適用して構想を削らない）。
   - **図の構想の尊重**: `- 図の構想:` ブロックがあるページは、構想を正として実文言化する。再発明・別形式への差し替えをしない（変えたくなったら Phase S に差し戻す）。
   - **主張と文言の規律**: [references/claims-and-language.md](references/claims-and-language.md) の全章を適用する — 増幅禁止（操作化チェック込み）・事実の書き分け・文言規範（ダッシュ禁止・ページ参照の用途規則・引用の正規化）・**文言スロップの負のリスト**（呼びかけ疑問文・共感定型・効能の約束・登壇口調。S の方針より優先）。数字・固有名は出所を1件ずつ資料と照合する。
4. **自己検証**:
   - (a) takeaway だけを1→Nで通読し、紙面上のメッセージ連鎖がゴールに到達するか（ゴーストデッキの最終形）
   - (b) 増幅チェック（全称・排他・最上級の機械抽出→資料該当行の引用照合）
   - (c) 文言発明ゼロテスト（実装時に発明が必要な文言・数字・ラベルが残っていないか）
   - (d) 空間予算超過の検出
   - (e) 用語・比喩・ラベルの一貫性 — 同一概念に別の比喩・別の呼び名を使わない（例: 同じ仕組みをあるページで「USBポート」、別ページで「電源プラグ」とたとえない）。先出しの語彙・役割名は後のページで再利用する。**工程・サイクル等の順序と番号も再掲時に一致させる**（同じサイクルを別ページで別の開始点・別の番号で回さない — 循環として等価でも、読者には別の手順に見える）
   - (f) ノイズゲート（[references/claims-and-language.md](references/claims-and-language.md) 6章。機能を1行で言えない要素は削る)
   - (g) 文言スロップ点検（導入・まとめページを中心に負のリスト照合）
   - (h) **図の含意チェック** — 図の構造（中心性・包含・大きさ・線の本数・方向）が、文言の支持しない主張をしていないか図ごとに確認する。図は文言と別のチャネルで主張する（例: 規格・概念を中央の単一箱で描くと「中央サービスがある」と読まれ、本文と矛盾しうる）。増幅チェックと同じ基準を視覚構造にも適用する。
5. 肉付け済み storyboard.md を提示し、**ターンを終えて合意を待つ**。文言はここがユーザの最終確認点（実装後の文言修正は肉付けを先に更新してから反映する）。

**このフェーズで決まらないこと（実装層の責務）**: レイアウト・配色・タイポグラフィ・コンポーネント選択・厳密な配置。

## Phase I: 実装（1ページずつ・インタラクティブ）

**目的**: 肉付けを Slidev の紙面に置く。文言・数字・ラベルは肉付けが正で転記のみ。実装の裁量はレイアウト・コンポーネント選択・配置・視覚形式の実現方法。**1ページずつ実装→レンダ→提示→修正で完成させ、まとめて作ってまとめて直すことをしない** — ページ毎なら初期の指摘を後続ページへ先回り適用できる。

**手順**:

0. **Scaffold**: テンプレート一式をデッキプロジェクトへ展開し、Phase T 承認済みの theme.css とフレームプリセットを適用する。deckMode はブリーフの用途から設定する（登壇=presentation、配布・報告=reading）。骨格レンダの確認は `npx slidev export --format png --range 1` で行い、specimen と実レンダの差をここで一度検分する。開発サーバの常駐は不要 — 起動した場合は作業後に停止し、**プロジェクト外に設定ファイルを残さない**。
1. **部品の棚卸し→ページ実装**: 実装前に [references/components.md](references/components.md) とデッキの style-notes.md を棚卸しする。**ページ型部品**（IntroToc・BigStatement・RecapList・FigurePage・FullVisual）に該当する型のページは必ず部品で実装し、生HTMLで組み直さない。**要素部品**（Badge・FlowSteps・Compare・KeyNumber・LeadBox・ChatExample 等）は該当があれば流用し、自作は部品が無い場合のみ。ページ N を肉付けブロックどおりに実装する（順序は表紙→1→…→N。図は肉付け・図の構想の忠実な視覚化であり、要素・ラベルの取捨は行わない）。表紙は肉付けの対象外で、文言は deck-brief のテーマ欄から転記する（タイトル・サブタイトルへの分割は配置判断として可）。author・date 等のテンプレート欄は、ブリーフ・入力資料に値が無ければ欄ごと消す。
2. 区切りごとに lint（`scripts/lint-deck.mjs`）を実行する。
3. 当該ページだけ PNG に書き出し（`npx slidev export --format png --range N --output slides-export/page-N`。**出力先は必ずページ別に分ける** — export は出力ディレクトリを毎回作り直すため、共有すると過去ページの検品用PNGが消える）、**まず自分で検分する**: 崩れ（はみ出し・重なり・文字色の白落ち）→ [references/aesthetics.md](references/aesthetics.md) 8章の**検分チェックリスト14問**を上から適用。不合格なら直してから提示する。実装中の見た目の判断基準は同ファイル1〜7章。
4. PNG をユーザに提示し、**ターンを終えて反応を待つ（ページ毎の確認ループ）**。OK で次ページへ。修正指示があれば直して再提示。
5. **修正指示を個別ページで終わらせない**: 後続ページにも効く指示は `style-notes.md` に追記し、以後の全ページへ先回り適用する。既済ページに波及する指示なら該当ページも直す（同じ指摘を2度させない）。ゲート指示由来に限らず、**実装中に自分で確立した取り決め**（表記の正規化・視覚文法・部品の流用方法）も style-notes.md へ記録する。テンプレート由来のファイル（scripts・components・layouts・specimen 等）に不具合修正を加えた場合は、デッキ内で直して終わりにせず、**template還流の申し送り**として完成報告に列挙する（修正はコピーに閉じており、次のデッキは原本から作られる。還流しないと全デッキが同じ不具合に再遭遇する）。
6. 同じ視覚パターンが2回目に出たら初回の実装を再利用する。
7. **納品書き出し**: 全ページ OK 後、最終成果物として **PDF と pptx を必ず書き出す**（`npx slidev export --format pdf --output slides-export/final/<デッキ名>.pdf` と `npx slidev export --format pptx --output slides-export/final/<デッキ名>.pptx`。pptx は各ページを画像スライドとして埋め込む形式で文字の再編集は不可 — 配布・投影用）。書き出し後、ファイルの存在とスライド枚数がデッキと一致することを確認してから、完成報告（template還流の申し送りを含む）とともに提出する。ユーザは全ページを目視確認済みのため、以降の検証（通し lint・最終ゴースト確認・伝達テスト deck-transmission-test）は**任意の校正オプション**であり必須ではない。実施するかをユーザに確認する。
